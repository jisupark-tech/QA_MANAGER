# CS/운영 업무 AI 대체 기획안

> **작성일**: 2026-02-27
> **목적**: 운영/CS 업무를 AI로 자동화하는 3가지 접근법 비교 및 권장 방안 제시

---

## 1. 배경 및 문제 정의

### 현재 상황
- 운영/CS 업무는 반복적이고 패턴화된 작업이 대부분
- 인력 의존도가 높아 확장성에 한계
- 응답 시간, 일관성, 24/7 대응 등에서 개선 여지 존재

### 핵심 질문
> "운영/CS 업무를 Moltbook이나 Claude Code/Cowork로 대체할 수 없나?"

### 결론 요약
| 접근법 | 적합도 | 핵심 장단점 |
|--------|--------|-------------|
| A. Claude Code/Cowork 직접 사용 | **낮음** | 개발자 도구를 CS 담당자가 직접 쓰기 어려움 |
| B. Claude Code로 CS/운영 AI 제작 | **높음 (권장)** | 맞춤형 AI 에이전트 → 우리 업무에 최적화 |
| C. Moltbook 활용 | **중간** | 세팅 후 자체 실행 가능, 커스터마이징 한계 |

## 2. 세 가지 접근법 상세 비교

### 접근법 A: Claude Code/Cowork 직접 사용

```
CS 담당자 → Claude Code CLI 또는 Cowork UI → 업무 처리
```

**장점**
- 별도 개발 불필요, 즉시 사용 가능
- Claude의 높은 추론 능력 활용

**단점**
- CS 담당자가 프롬프트 엔지니어링을 해야 함
- 업무 프로세스가 표준화되지 않음 (사람마다 다른 질문 → 다른 결과)
- 사내 데이터(FAQ, 정책, 이력)와의 연동이 수동적
- 매번 컨텍스트를 새로 입력해야 함

**적합 상황**: 개발팀 내부의 임시 CS 대응, 소규모 프로젝트


### 접근법 B: Claude Code로 CS/운영 AI 에이전트 제작 (권장)

```
개발자 → Claude Code로 CS AI 시스템 구축
                ↓
CS 담당자/유저 → 완성된 CS AI → 자동 응답/처리
```

**장점**
- 우리 서비스에 맞는 맞춤형 AI 에이전트
- FAQ, 정책, 유저 데이터 DB 연동 자동화
- 응답 품질 일관성 보장 (프롬프트 + 가드레일 내장)
- 에스컬레이션 규칙 자동 적용
- 한번 만들면 유지보수만 필요 (확장성 높음)

**단점**
- 초기 개발 기간 필요 (2~4주)
- 개발 인력 투입 필요

**적합 상황**: 중장기 운영, 대규모 CS, 자사 서비스 맞춤 대응


### 접근법 C: Moltbook 활용

```
관리자 → Moltbook 세팅 (API Key + 워크플로우) → 자체 실행
```

**장점**
- 코딩 없이 워크플로우 구성 가능
- API Key만 넣으면 자체적으로 실행
- 노코드/로코드 접근으로 비개발자도 세팅 가능

**단점**
- 초기 세팅에 시간 소요
- 플랫폼 종속 (Moltbook 가격/정책 변경 리스크)
- 깊은 커스터마이징 한계 (복잡한 분기 처리 등)
- 자사 DB와의 실시간 연동에 제약 가능

**적합 상황**: 빠른 도입, 표준적인 CS 업무, 개발 리소스 부족 시

## 3. 권장 방안: 하이브리드 전략 (B + C)

### 전략 개요

```
┌─────────────────────────────────────────────────────┐
│                  CS/운영 AI 시스템                     │
├──────────────────────┬──────────────────────────────┤
│   즉시 도입 (1~2주)   │     맞춤 구축 (3~6주)         │
│                      │                              │
│   Moltbook           │   Claude Code 기반            │
│   ├ 단순 FAQ 응답     │   CS AI 에이전트              │
│   ├ 템플릿 기반 안내   │   ├ 유저 DB 연동 자동 응답     │
│   └ 기본 티켓 분류     │   ├ 복합 문의 추론 처리        │
│                      │   ├ 자동 에스컬레이션          │
│                      │   ├ 운영 대시보드 연동          │
│                      │   └ 밸런스/버그 리포트 자동 분석 │
├──────────────────────┴──────────────────────────────┤
│                  공통 인프라                           │
│   ├ FAQ/정책 문서 DB (벡터 검색)                       │
│   ├ 유저 이력 DB (결제, 문의, 제재)                    │
│   ├ 에스컬레이션 규칙 엔진                             │
│   └ 응답 품질 모니터링 + 피드백 루프                    │
└─────────────────────────────────────────────────────┘
```

### Phase 1: Moltbook 즉시 도입 (Week 1~2)
| 항목 | 내용 |
|------|------|
| 대상 업무 | 단순 FAQ, 정형화된 안내, 기본 티켓 분류 |
| 세팅 내용 | API Key 연결, FAQ 문서 업로드, 응답 템플릿 구성 |
| 기대 효과 | 단순 문의 60~70% 자동 처리 |
| 필요 인력 | 운영 담당자 1명 (세팅) |

### Phase 2: CS AI 에이전트 설계 (Week 2~3)
| 항목 | 내용 |
|------|------|
| 대상 업무 | 복합 문의, 유저 데이터 조회 필요한 건, 운영 판단 필요한 건 |
| 설계 내용 | 아래 섹션 4 참조 |
| 필요 인력 | 개발자 1명 + 운영 담당자 1명 (요구사항 정리) |

### Phase 3: CS AI 에이전트 구축 (Week 3~6)
| 항목 | 내용 |
|------|------|
| 구현 | Claude API 기반 에이전트 + DB 연동 + 관리 UI |
| 테스트 | 과거 CS 로그로 정확도 검증 (목표: 85%+) |
| 배포 | Moltbook 단순 응답 → CS AI 복합 응답 → 사람 에스컬레이션 |

### Phase 4: 통합 및 고도화 (Week 6~8)
| 항목 | 내용 |
|------|------|
| Moltbook 역할 축소 | 단순 라우팅/1차 필터로 전환 |
| CS AI 확장 | 운영 업무 자동화 (밸런스 리포트, 이상 탐지 등) |
| 피드백 루프 | 미해결 건 분석 → AI 개선 → 정확도 상향 |

## 4. CS AI 에이전트 설계 (접근법 B 상세)

### 시스템 아키텍처

```
유저 문의 (채널: 인앱/이메일/디스코드/카카오톡)
         │
         ▼
  ┌─────────────┐
  │  Gateway     │  ← 채널 통합, 메시지 정규화
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │  Classifier  │  ← 문의 유형 분류 (LLM)
  │  (분류기)     │     카테고리: FAQ / 계정 / 결제 / 버그 / 건의 / 제재
  └──────┬──────┘
         ▼
  ┌─────────────────────────────────────────┐
  │              Router (라우터)              │
  ├────────┬────────┬────────┬──────────────┤
  │  FAQ   │ 계정/  │ 결제   │   복합/      │
  │  Agent │ 제재   │ Agent  │   에스컬     │
  │        │ Agent  │        │   레이션     │
  ├────────┼────────┼────────┼──────────────┤
  │ 벡터DB │ 유저DB │ 결제DB │  사람 CS     │
  │ 검색   │ 조회   │ 조회   │  담당자에게  │
  │ → 응답 │ → 응답 │ → 응답 │  전달        │
  └────────┴────────┴────────┴──────────────┘
         │
         ▼
  ┌─────────────┐
  │  QA Guard    │  ← 응답 검증 (민감정보 마스킹, 톤 체크)
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │  Responder   │  ← 최종 응답 전송 + 로깅
  └─────────────┘
```

### 주요 모듈

#### 4-1. FAQ Agent
- **입력**: 유저 질문
- **처리**: FAQ 벡터 DB 검색 → 유사도 상위 3건 참조 → LLM이 자연어 응답 생성
- **DB**: FAQ 문서, 공지사항, 패치노트, 가이드
- **자동화율 목표**: 단순 문의의 80%

#### 4-2. 계정/제재 Agent
- **입력**: 유저 ID + 문의 내용
- **처리**: 유저 DB 조회 (접속 이력, 제재 이력, 보유 자산) → 상황 분석 → 응답/조치
- **권한 레벨**:
  - Level 1 (자동): 정보 조회 + 안내 (로그인 방법, 연동 확인)
  - Level 2 (반자동): 아이템 복구, 임시 제재 해제 → 담당자 승인 후 실행
  - Level 3 (사람): 영구 제재, 대규모 보상 → 에스컬레이션

#### 4-3. 결제 Agent
- **입력**: 유저 ID + 결제 관련 문의
- **처리**: 결제 DB 조회 → 영수증 대조 → 미지급 확인 → 자동 지급 또는 에스컬레이션
- **가드레일**: 금액 상한선 초과 시 반드시 사람 검수

#### 4-4. 운영 자동화 모듈 (CS 외 확장)
| 기능 | 설명 |
|------|------|
| 밸런스 이상 탐지 | 유저 데이터 통계 → 비정상 패턴 알림 |
| 버그 리포트 집계 | 유사 버그 자동 그룹핑 → 개발팀 리포트 생성 |
| 공지/이벤트 초안 | 운영 지시 → 공지문 자동 작성 (검수 후 게시) |
| KPI 일일 리포트 | DAU, 매출, 리텐션 데이터 → 요약 리포트 자동 생성 |

### 에스컬레이션 규칙

```yaml
escalation_rules:
  - condition: "결제 금액 > 100,000원"
    action: human_review
    priority: high

  - condition: "유저 감정 = 분노 (LLM 판정)"
    action: human_takeover
    priority: urgent

  - condition: "AI 확신도 < 0.7"
    action: human_review
    priority: medium

  - condition: "법적 언급 (소송, 고소, 법률)"
    action: human_takeover
    priority: urgent

  - condition: "동일 유저 3회 이상 재문의"
    action: human_takeover
    priority: high
```

### 기술 스택

| 컴포넌트 | 기술 | 비고 |
|----------|------|------|
| LLM | Claude API (Sonnet) | 비용 효율 + 충분한 성능 |
| 벡터 DB | Pinecone 또는 Chroma | FAQ 검색용 |
| 백엔드 | Node.js / Python FastAPI | 채널 게이트웨이 + 라우터 |
| 유저/결제 DB | 기존 서비스 DB 연동 | Read-only 접근 |
| 관리 UI | 간단한 웹 대시보드 | 에스컬레이션 큐, 응답 로그, 통계 |
| 채널 연동 | 카카오톡 API, Discord Bot 등 | 서비스에 맞게 선택 |

## 5. 비용 분석

### 인건비 vs AI 비용 비교 (월 기준)

| 항목 | 현재 (사람) | AI 도입 후 |
|------|------------|-----------|
| CS 인력 | 2~3명 × 월급 | 0.5~1명 (에스컬레이션 처리) |
| AI API 비용 | - | 월 $200~500 (문의량 따라) |
| Moltbook 비용 | - | 월 구독료 (플랜별) |
| 개발 비용 | - | 초기 1회: 개발자 1명 × 4~6주 |

### 예상 ROI
- **손익 분기점**: 도입 후 2~3개월
- **1년 누적 절감**: 인건비 기준 50~70% 절감 예상
- **부가 가치**: 24/7 대응, 응답 시간 단축 (평균 수 시간 → 수 초), 일관된 품질

|--------|------|-----------|
| AI 오답/환각 | 유저 불만, 신뢰도 하락 | QA Guard + 확신도 기반 에스컬레이션 |

| 복잡한 문의 | AI 한계 도달 | 에스컬레이션 규칙으로 빠른 사람 전환 |
